# README.md — Elody-Farm Loyalty System

## 1. Название проекта
**Elody-Farm Loyalty** — мобильная система лояльности для аптечной сети **Elody-Farm** с цифровой дисконтной картой, балансом бонусов, историей транзакций, детализацией покупок, акциями, уведомлениями, анкетами и каталогом аптек на карте.

---

## 2. Цель проекта
Цель проекта — повысить удержание клиентов аптечной сети Elody-Farm за счёт удобного мобильного приложения, которое:

- заменяет физическую карту лояльности цифровой;
- показывает бонусный баланс и доступные скидки;
- даёт прозрачную историю начислений и списаний;
- позволяет видеть состав каждой покупки;
- показывает актуальные акции и персональные предложения;
- помогает находить ближайшие аптеки;
- создаёт постоянный канал коммуникации с клиентом.

Проект должен быть спроектирован так, чтобы его могли реализовать **AI-агенты без участия программистов**, поэтому это README описывает не только продукт, но и пошаговую схему проектирования, разработки и контроля качества.

---

## 3. Общая концепция продукта
Приложение должно быть простым, понятным и визуально чистым. Основной сценарий пользователя:

1. Пользователь вводит номер телефона.
2. Входит по паролю или восстанавливает доступ через SMS-код.
3. Видит главный экран с балансом и акциями.
4. Открывает цифровую карту лояльности со штрихкодом.
5. Отслеживает бонусы, транзакции и покупки.
6. Просматривает акции, уведомления и опросы.
7. Редактирует профиль, настройки и язык.
8. Находит ближайшую аптеку на карте.

На первом этапе backend может отсутствовать. Для этого mobile app должен уметь работать через **mock data** из локальных JSON-файлов.

- постоянные покупатели аптек Elody-Farm;

## 5. Технологический стек

### Backend
- **Python**
- **Django**
- **Django Ninja** для REST API
- PostgreSQL
- Redis для кэшей, rate limit, задач и OTP
- Celery для фоновых задач
- JWT / session token для авторизации

### Mobile Frontend
- **Flutter**
- Dart
- Архитектура: Feature-first + Clean-ish layers
- State management: `Riverpod` или `Bloc` (рекомендуется Riverpod как более простой для AI-генерации)
- Локальное хранение: `shared_preferences` / `flutter_secure_storage`
- HTTP: `dio`
- Карты: `google_maps_flutter`
- Генерация штрихкода: `barcode_widget`
- Локализация: `flutter_localizations`, `intl`

### Дизайн и прототипирование
- Figma
- Material 3 как база, но с кастомной темой Elody-Farm

## 6. Режимы разработки по этапам
Проект нужно строить поэтапно.

### Этап 1. Discovery и формализация требований
Нужно:
- описать все пользовательские роли;
- описать все экраны;
- описать пользовательские сценарии;
- определить сущности данных;
- определить MVP и post-MVP.

### Этап 2. UX/UI и проектирование экранов
Нужно:
- создать карту экранов;
- описать каждый экран;
- перечислить виджеты;
- определить состояния экранов: loading, empty, error, success.

### Этап 3. Mobile MVP на mock data
Нужно:
- собрать Flutter-приложение без backend;
- подключить локальные JSON-файлы;
- реализовать навигацию;
- реализовать визуально все ключевые экраны;
- показать жизнеспособность продукта.

### Этап 4. Проектирование backend
Нужно:
- определить модели данных Django;
- спроектировать API endpoints;
- определить схему авторизации;
- определить контракты между mobile и backend.

### Этап 5. Интеграция mobile + backend
Нужно:
- заменить mock-репозитории на API-репозитории;
- внедрить токены;
- обработать ошибки сервера;
- включить реальные транзакции, акции и аптеки.

### Этап 6. Тестирование и релиз
Нужно:
- unit tests;
- widget tests;
- integration tests;
- smoke test сценариев;
- сборки Android / iOS.

## 8. Анализ предоставленных референсов и выводы для ТЗ
По скриншотам видно несколько важных паттернов, которые стоит сохранить и адаптировать под Elody-Farm.

### 8.1 Экран авторизации
На скриншотах:
- вход начинается с выбора страны и ввода телефона;
- используется большой приветственный заголовок;
- поля занимают почти всю ширину;
- CTA-кнопка крупная и закреплена в нижней зоне;
- после ввода телефона появляется экран пароля;
- есть сценарий восстановления пароля;
- подтверждение кода сделано через 4 отдельных ячейки.

**Рекомендация для Elody-Farm:**
- оставить сценарий «телефон → пароль / восстановление»;
- использовать крупный заголовок и минималистичную форму;
- добавить маску номера для Молдовы по умолчанию;
- оставить выбор страны, но основным кейсом сделать `+373`;
- предусмотреть OTP-экран из 4 или 6 ячеек.

### 8.2 Главный экран
На скриншотах:
- сверху блок баланса;
- ниже горизонтальная карусель акций;
- снизу нижняя навигация с 3 вкладками;
- дизайн чистый, с большим количеством белого пространства.

**Рекомендация:**
- на home показывать баланс, доступный кэшбэк, ближайшую аптеку, 3–5 активных акций и быстрые действия;
- нижняя навигация: `Главная / Карта / Профиль` либо `Главная / Карта / Профиль`, а экран цифровой карты сделать центральной плавающей кнопкой.

### 8.3 Экран акций и новостей
На скриншотах:
- вкладки `Акции / Новости`;
- карточки с большими изображениями;
- есть empty-state экран.

**Рекомендация:**
- реализовать один общий модуль `Promotions`, где есть табы: `Акции`, `Новости`;
- карточки должны поддерживать баннер, дату, название, короткое описание и переход в детали;
- обязательно предусмотреть empty state.

### 8.4 Цифровая карта
На скриншотах:
- есть большая карточка со штрихкодом;
- рядом свайп для добавления физической карты;
- под картой отображаются баланс и доступно;
- ниже пустое состояние транзакций.

**Рекомендация:**
- цифровая карта должна быть центральным экраном продукта;
- поддержать две сущности: `digital card` и `physical card`;
- добавить состояние, когда физическая карта ещё не привязана;
- под картой показывать summary: `всего начислено`, `доступно`, `сгорело`, `потрачено`.

### 8.5 Профиль и редактирование
На скриншотах:
- профиль содержит имя и телефон;
- есть редактирование персональных данных;
- можно загрузить фото;
- указывается дата рождения и пол.

**Рекомендация:**
- поля профиля: имя, фамилия, отчество, дата рождения, пол, телефон, email, фото;
- дата рождения нужна для персональных акций и бонуса ко дню рождения.

### 8.6 Аптеки на карте
На скриншоте:
- карта с пинами;
- есть поиск;
- есть список и карта одновременно.

**Рекомендация:**
- поддержать режимы `Карта` и `Список`;
- на карточке аптеки показывать адрес, часы работы, дистанцию, телефон, доступные услуги.

### 8.7 Уведомления и опросы
На скриншотах:
- уведомления и опросы разделены табами;
- оба экрана поддерживают пустые состояния.

**Рекомендация:**
- сделать единый экран `Коммуникации` с вкладками `Уведомления` и `Опросы`;
- пустые состояния и счётчики непрочитанных обязательно поддержать.

### 8.8 Настройки
На скриншотах:
- язык приложения;
- уведомления;
- обратная связь;
- правила программы;
- информация о приложении;
- информация о разработчике;
- изменить пароль;
- удалить аккаунт;
- выйти.

**Рекомендация:**
- для Elody-Farm этот список подходит почти полностью;
- дополнительно можно добавить `Политика конфиденциальности` и `Согласие на маркетинговые уведомления`.

## 10. Детальное ТЗ на мобильное приложение

# 10.1 Экран: Welcome / Ввод телефона
### Назначение
Точка входа в приложение.

### Элементы
- заголовок: `Добро пожаловать` или `Приветствуем`;
- поле выбора страны с флагом и кодом;
- поле ввода номера телефона;
- кнопка `Далее`;
- опционально ссылка на пользовательское соглашение.

### Логика
- при старте по умолчанию страна = Moldova (`+373`);
- поле ввода должно поддерживать маску;
- кнопка активируется только при валидном телефоне.

### Валидации
- пустое поле;
- слишком короткий номер;
- недопустимые символы.

### Состояния
- default;
- invalid;
- loading после нажатия `Далее`.

# 10.2 Экран: Ввод пароля
### Элементы
- телефон пользователя;
- поле пароля;
- иконка показать/скрыть пароль;
- кнопка `Войти`;
- ссылка `Забыли пароль?`.

### Логика
- если номер зарегистрирован, открывается этот экран;
- при успешном входе переход на home;
- при ошибке показывается текст ошибки под полем.

- неверный пароль;

- временно недоступен сервер.

## 19. Требования к качеству кода для AI-генерации
Чтобы AI-агенты генерировали код стабильно, нужно придерживаться правил:

- один экран = одна feature-папка;
- не смешивать UI и доступ к данным;
- все модели типизированы;
- все JSON имеют явные DTO-модели;
- каждая фича имеет repository interface;
- не использовать магические строки;
- все константы вынести;
- все тексты вынести в локализацию.

### Для Flutter
- включить `flutter_lints`;
- использовать `freezed` и `json_serializable`, если AI-агент справится стабильно;
- если нет, допустимы простые ручные модели, но единообразные.

### Для Python
- использовать typing;
- pydantic schemas через Ninja Schema;
- разделять service layer и API layer.

## 20. Acceptance Criteria для mobile MVP
MVP считается готовым, если:

1. Пользователь может пройти сценарий `ввод телефона → пароль → home`.
2. Пользователь может пройти сценарий восстановления пароля.
3. На home отображается mock баланс и mock акции.
4. Цифровая карта отображает штрихкод и номер карты.
5. История транзакций открывается и показывает mock данные.
6. Можно открыть детали транзакции с товарами.
7. Профиль открывается и редактируется локально.
8. Экран аптек показывает карту и список mock аптек.
9. Акции и новости имеют список и empty state.
10. Уведомления и опросы имеют список и empty state.
11. Приложение переключается минимум между русским и румынским языком.
12. Все критические экраны имеют loading/empty/error states.

## 22. Риски и ограничения

### Риски
- отсутствие backend на старте может привести к переделкам контрактов;
- если mock-модели не совпадут с будущим API, интеграция усложнится;
- слишком сложная архитектура затруднит генерацию AI-агентами.

### Как снизить риск
- сразу зафиксировать DTO и repository interfaces;
- использовать mock-first, но contract-driven подход;
- держать README источником истины.
